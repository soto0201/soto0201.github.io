久しぶりのハリーズでした

先週までは

自粛でしばらく研究室に寄っていなかったり、そもそもメンタルをやっていたというのもあり、
今日は久しぶりにハリーズのカレーを食べました。美味しかったです。
先週まではたけ田商店の唐揚げをよく食べていました。これも美味しいです。

私が運転をしてハリーズまで行ったのですが、若干遠く往復で1時間ほどです。
16時頃に注文してカレーを受け取りに行くのですが、帰りは学生の下校時間と被ってしまうので結構時間がかかります…
その時間があればゼミで発表する進捗をまとめられる…と思いつつも食欲には勝てないので行きます。

今回のゼミ

先週は先輩方がOS研究会の発表でした。今年はB4の参加は無いです。
申し込みが間に合えばTobiuo氏も発表できていたはずなのですが、まあそれはそれです…
先輩方お疲れ様でした。

そして何故か私はTobiuo氏が今やっていることを簡単になら説明できます。
去年Koo先輩がやっていたことをまずは再現してみるというものです。
話しを聞いていて個人的に面白そうだったので軽く論文を読んでしまいました。

研究内容を説明すると、大前提としてPerl6のコンパイルが遅いことが問題として取り上げられています。
その理由はコンパイルする際に内部でVM(MoarVM)が立ち上がりますが、それがコンパイルの遅さの原因となっています。
そこで、サーバーを立ち上げておきVMを起動しっぱなしにしたらネックとなる部分が解消されるのではないかというものです。

一見簡単そうに見えます。しかしそうスムーズに物事は進んでくれません。
論文と一緒にコードがありますが、Perl6はその時点よりバージョンが上がっているために動きません。
そのために少し面倒な事になっているようです。

ところで自分の進捗は？

ブログでしきりにいっていましたが、論文を翻訳しています。ついに3章まで読み終えました。
Hoare Logic の部分的正当性のところまでは読んだ感じです。そこまで理解ができると研究目的もかなり書けました。

専門用語の説明を後述しておきます…
Hoare LogicとCbCについてです。用語がわからない場合はそちらから読んだ方が理解がしやすいかもしれません…

私の研究内容はHoare Logicを用いたCbCの検証になると思っています。
そしてHoare LogicをCbCに当てはめると、以下のようになると思っていました。

引数となるDatasegmentをP、CodesegmentをQ、実行後のDatasegmentをRとできる。
これを検証することでそのCodesegmentが正当であることが示せる。

しかし、これでは検証には足りないようです。

足りない部分というのは、以下2点です。

  • プログラムの開始時にinput(入力)を行うことがあるが、これの検証方法の説明が無い。
  • プログラムが停止する際の説明が無い。

これらが足りないのは、ryokkaさんの論文から感じ取っていました。
しかし、論文を読み切れていなかったため、その部分の説明ができないので書くことができませんでした…

そしてPre ConditionとPost Conditionがこれらの検証にに必要とのアドバイスをkono先生からもらいました。
ここら辺は何もわからないのですが、ryokkaさんの修士発表ポスターでは急に発生していた単語でした。
ですので、今日はこれが何に使うものであるのか知ることができただけでも嬉しい収穫です。

Hoare Logicの部分正当性とは

プログラム(Q)の開始前に事前条件(P)が真の場合、
結果(R)はその完了時に真となるというもの。

こちらは論文の内容をそのまま翻訳したものです。Hoare Logicの表面的な説明はこんな感じになります。
論文ではより詳細に説明しており、オーバーフローをした際の動きは考えない(検証に含めない？)ものとするなどがあります。

CbCとは

CbCとは、Cからループ制御構造とサブルーチンコールを取り除き、
継続を導入したCの下位言語である。継続呼び出しは引数付き goto 文で表現される。

上記の用件を満たすSegmentをCode Gearと呼び、そのCodeGearに引数として与えられる値をData Gearと読んでいます。
これらCode / Data Gearの継続によってCbCは構成されます。

明日は

明日はシス管の仕事で少し気になるところ(GitLabあたり)があったので、それを調べに学校に寄ろうかと思っています。
余裕があれば翻訳なども少し進めたいです。
